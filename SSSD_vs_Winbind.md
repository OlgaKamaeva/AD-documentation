Существует несколько способов подключения клиентских машин к домену Active Directory (AD):
1. Компонент Samba - **winbind**
2. **SSSD** (System Security Services Daemon, Демон служб безопасности системы), истоки которого лежат в проекте FreeIPA.

## SSSD
### Общая информация 
**SSSD** — это аббревиатура от System Security Services Daemon (демон системных сервисов безопасности). Это клиентская часть централизованных служб каталогов, таких как FreeIPA, Microsoft AD, Samba AD и других.

Клиент SSSD обслуживает и кэширует информацию, хранящуюся в службе каталогов, а также предоставляет сервисы идентификации, аутентификации и авторизации для локальной системы.

Помимо FreeIPA и Active Directory, SSSD может интегрироваться с другими службами каталогов и аутентификации:

+ Каталог LDAP – для взаимодействия с чистыми LDAP-серверами, которые предоставляют данные о пользователях и группах
+  Kerberos – для аутентификации через Kerberos, где вместо обычных паролей используются защищённые билеты (tickets)
+ локальную базу пользователей (files provider) – для работы с учетными записями, хранящимися в локальных файлах системы, например, в /etc/passwd и /etc/shadow

#### Аутентификация

Типы аутентификации в SSSD

SSSD поддерживает несколько типов аутентификации, включая:

+ Kerberos — для аутентификации через билеты Kerberos. Это используется, например, в интеграции с Active Directory (AD), где аутентификация выполняется через Kerberos, что обеспечивает высокий уровень безопасности, заменяя использование обычных паролей защищёнными билетом.

+ LDAP — для аутентификации через LDAP-серверы, такие как OpenLDAP или FreeIPA. В этом случае SSSD выполняет проверку подлинности пользователя с использованием LDAP-запросов к серверу каталога.

+ Local Authentication — для аутентификации с использованием локальных учетных записей, хранящихся в файлах /etc/passwd и /etc/shadow. Это используется, если SSSD настроен на работу с локальными пользователями.

+ Active Directory (AD) — при подключении к Active Directory используется комбинированный механизм с использованием как Kerberos, так и LDAP для аутентификации пользователей и получения информации из AD.

Для настройки аутентификации в SSSD, как правило, используется файл конфигурации /etc/sssd/sssd.conf. В нем определяются параметры подключения к серверу каталога (например, AD или LDAP), а также параметры аутентификации, такие как использование Kerberos или LDAP.

Пример настройки аутентификации для Active Directory:

	[domain/EXAMPLE.ALT]
	id_provider = ad
	auth_provider = ad
	chpass_provider = ad
	access_provider = ad
	default_shell = /bin/bash
	fallback_homedir = /home/%d/%u
	debug_level = 0
	; cache_credentials = false
	ad_gpo_ignore_unreadable = true
	ad_gpo_access_control = permissive
	ad_update_samba_machine_account_password = true

#### Авторизация

Авторизация в SSSD — это процесс, который происходит после успешной аутентификации пользователя и определяет, какие действия или ресурсы ему разрешены в системе. В отличие от аутентификации, которая подтверждает личность пользователя, авторизация отвечает на вопрос, имеет ли пользователь право доступа к определённым ресурсам или выполнить конкретные действия.

**Методы авторизации в SSSD**

SSSD позволяет управлять авторизацией пользователей и групп, используя:

+ Host-Based Access Control (HBAC) – это система, предоставляющая централизованный контроль доступа на основе хостов, которая используется, например, в FreeIPA.
+ Фильтры в конфигурации SSSD – В конфигурации SSSD /etc/sssd/sssd.conf можно использовать различные фильтры для авторизации пользователей и групп. Фильтры позволяют разрешать или запрещать доступ на основе атрибутов пользователей или групп, таких как имя, идентификатор группы или другие LDAP-атрибуты. permit - азрешает доступ всем пользователям (по умолчанию). deny - Блокирует всех пользователей. ldap - Проверяет доступ через LDAP-фильтр (например, членство в группе). ad - Использует правила безопасности AD, включая Group Policy Objects (GPO). hbac - Применяет Host-Based Access Control (HBAC) из FreeIPA.
	* SSSD проверяет access_provider, который определяет, можно ли пользователю войти в систему. 

**Механизм ID Mapping**

Для корректной авторизации между Linux-системой и службой каталогов, важно настроить ID mapping для отображения SID (Security Identifier) в UID и GID, что позволяет правильно работать с правами доступа для пользователей и групп из AD.

 По умолчанию пул UID/GID для сопоставления SID в SSSD имеет ограниченный размер. Для больших доменов с количеством пользователей более 200 тысяч этот пул необходимо расширять вручную.
 
+ При стандартной конфигурации настраивается 10000 срезов, каждый из которых может содержать до 200000 идентификаторов, начиная от 200000 и до 2000200000.
+ Из всего общего диапазона, размером 2 миллиарда под каждый домен выделяется срез id размером 200000, каждому домену может соответствовать только один единственный срез.
+ Для увеличения размера среза в конфигурации SSSD используются параметры:
	* ldap_idmap_range_min - нижняя (включительно) граница диапазона идентификаторов.
	* ldap_idmap_range_max - верхняя (не включительно) граница диапазона идентификаторов.
	* ldap_idmap_range_size - количество идентификаторов, доступных для каждого среза. Значение должно быть не меньше значения максимального RID пользователя, запланированного для использования на сервере AD.

Эти параметры позволяют адаптировать пул UID/GID под нужды вашего домена. Однако, увеличивая размер среза, необходимо уменьшать количество срезов, что увеличивает вероятность коллизий (по умолчанию вероятность коллизии одного конкретного домена с другим составляет 1/10000).

**Управление правами через SUDO**

Помимо контроля входа, SSSD может управлять sudo-правилами. Конфигурация sudo традиционно хранится в файле sudoers, который нужно вручную копировать и обновлять на каждой машине в системе. Однако в больших инфраструктурах это может быть неудобно и трудоемко, поэтому часто для упрощения управления правами доступа используют LDAP (Lightweight Directory Access Protocol), централизованный каталог для хранения конфигураций. В этом случае локальные системы просто обращаются к центральному серверу LDAP для получения актуальной информации о правилах доступа.

#### Автономный доступ 

Когда клиентская система использует SSSD для аутентификации с сервером каталога, все данные о пользователе и пароле могут быть закэшированы в локальной системе. Это значит, что если сервер каталога становится недоступен (например, из-за проблем с сетью), SSSD всё равно может предоставить доступ пользователю, используя данные из кэша.

#### Особенности
+ SSSD требует ручного указания настроек для каждого трастового домена в клиентской конфигурации в файле /etc/sssd/sssd.conf.
+ SSSD не поддерживает трасты уровня леса (Forest Trusts), что ограничивает его возможности при работе с многоуровневыми лесами доменов. Однако, для большинства стандартных трастов (External Trust) SSSD может быть применен.
	+ SSSD использует Winbind, поэтому на клиенте AD-домена Winbind должен быть установлен в любом случае.
## Winbind
### Общая информация
Winbind — это компонент Samba, предназначенный для интеграции Linux-систем с доменами Active Directory (AD). Он использует Unix-реализацию Microsoft RPC-вызовов, модульную систему аутентификации (PAM) и Name Service Switch (NSS), позволяя Linux клиентам стать полноправными членами домена Active Directory (как MS AD, так и Samba AD).

Winbind позволяет:

+ Подключить Linux машину к домену Active Directory.
+ Управлять аутентификацией и авторизацией пользователей.
+ Взаимодействовать с учетными записями и группами, хранящимися в AD.
+ Обрабатывать запросы на групповую политику (GPO), если это необходимо.

#### Аутентификация
Winbind поддерживает несколько типов аутентификации, включая:

+ Kerberos — для аутентификации через Kerberos, стандартный протокол аутентификации в Active Directory. Winbind использует Kerberos для получения билетов доступа и взаимодействия с AD для аутентификации пользователей.

+ NTLM — в случае, если Kerberos недоступен или не используется, Winbind может прибегать к аутентификации через NTLM (NT LAN Manager), который является старым, но все еще поддерживаемым методом аутентификации в Windows-средах.

+ LDAP — Winbind использует LDAP для получения информации о пользователях и группах из Active Directory. Это важно для получения информации о группах пользователей, их членствах и атрибутах.

#### Авторизация

**Механизм ID Mapping**

Домены Windows различают пользователей и группы по уникальным идентификаторам безопасности (SID). Однако в Linux для каждого пользователя и группы требуются уникальные идентификаторы UID и GID. Служба winbindd отвечает за предоставление информации о пользователях и группах домена.

Чтобы служба winbindd могла предоставлять уникальные идентификаторы для пользователей и групп в Linux, необходимо на клиенте домена настроить сопоставление идентификаторов в файле /etc/samba/smb.conf для:

+ Локальная база данных (домен * по умолчанию)
+ Домен AD
* Каждый доверенный домен, из которого пользователи должны иметь доступ к ресурсам

**Пример конфигурации для Winbind с AD**

Для настройки интеграции с AD с использованием Winbind необходимо настроить несколько параметров в конфигурационном файле Samba (/etc/samba/smb.conf), а также убедиться, что сервисы Kerberos и Winbind настроены правильно.

Пример конфигурации для подключения к Active Directory:

	[global]
        log level = 10
        security = ads
        realm = EXAMPLE.ALT
        workgroup = EXAMPLE
        netbios name = WSTEST
        template shell = /bin/bash
        kerberos method = system keytab
        wins support = no
        winbind use default domain = yes
        winbind enum users = no
        winbind enum groups = no
        template homedir = /home/EXAMPLE.ALT/%U
        winbind refresh tickets = yes
        winbind offline logon = yes
        idmap config * : range = 10000-20000000
        idmap config * : backend = tdb

#### Особенности 
+ Winbind не поддерживает возможность динамического обновления DNS-записей. Для обхода этой проблемы была разработана программа, реализующая динамическое обновление адресов на DNS-сервере при использовании winbind в качестве клиента домена — winbind-dnsupdate. Подробности...
+ Winbind использует вызовы MSRPC для запросов информации о пользователях и группах домена, а также для аутентификации и изменении паролей. 
+ Winbind предназначен для работы именно как клиент Active Directory и не умеет работать с другими службами каталогов и аутентификации.
+ Winbind поддерживает NTLM-аутентификацию.
